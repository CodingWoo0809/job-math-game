# 직무수학 어드벤처 게임 제작 워크플로

> 문서 상태: Prototype v0.1  
> 기준 문서: `agent.md`  
> 이 문서는 프로젝트가 진행되면서 계속 갱신한다.

## 1. 목적

이 문서는 직무수학 어드벤처 게임을 기획하고, 구현하고, 검증하고, 배포하는 순서를 정의한다.

핵심 목표는 18개 맵을 빠르게 채우는 것이 아니라 다음 세 요소가 끝까지 함께 작동하도록 만드는 것이다.

- **교육**: 성취기준에 맞는 수학적 사고가 일어난다.
- **게임**: 플레이어의 관찰과 선택이 세계를 변화시킨다.
- **이야기**: WH가 집에서 학교까지 이동하며 수학의 필요성과 유용성을 발견한다.

## 2. 작업의 기본 원칙

1. 모든 작업은 `agent.md`를 기준으로 한다.
2. 교육과정 설계와 게임 설계를 분리하지 않고 동시에 검토한다.
3. 18개 맵 전체를 구현하기 전에 대표 맵 하나를 처음부터 끝까지 플레이 가능한 상태로 만든다.
4. 큰 기능보다 작고 검증 가능한 단위로 제작한다.
5. 새 기능을 추가할 때 기존 맵의 진행과 판정을 다시 확인한다.
6. 결정된 내용, 가정, 검증이 필요한 내용은 문서에 남긴다.
7. 완성되지 않은 임시 요소는 `임시`, 교육적 근거가 확인되지 않은 내용은 `검증 필요`로 표시한다.

## 3. 챕터 및 맵 구조

전체 게임은 직무수학의 4개 단원에 대응하는 **4개 챕터, 총 18개 맵**으로 구성한다. 성취기준 수에 맞추어 챕터 1에는 3개 맵, 챕터 2~4에는 각 5개 맵을 배치한다.

| 챕터 | 임시 챕터명 | 관련 단원 | 맵 범위 | 주 성취기준 |
|---|---|---|---|---|
| 챕터 1 | 수와 연산 | 01. 수와 연산 | 1map1~1map3 | [12직수01-01]~[12직수01-03] |
| 챕터 2 | 변화와 관계 | 02. 변화와 관계 | 2map1~2map5 | [12직수02-01]~[12직수02-05] |
| 챕터 3 | 도형과 측정 | 03. 도형과 측정 | 3map1~3map5 | [12직수03-01]~[12직수03-05] |
| 챕터 4 | 자료와 가능성 | 04. 자료와 가능성 | 4map1~4map5 | [12직수04-01]~[12직수04-05] |

현재 챕터명은 구분을 위한 **임시 이름**이다. 각 챕터의 장소, 사건과 분위기가 정해지면 단원명을 그대로 사용하지 않고 게임 세계에 어울리는 이야기형 이름으로 변경한다. 변경한 이름만 보아도 챕터의 분위기와 목표는 느껴지되, 학습 단원과의 연결은 기획 문서에서 계속 추적할 수 있어야 한다.

### 챕터 설계 원칙

- 각 맵은 해당 단원의 성취기준 하나를 중심으로 구성한다. 챕터 1은 3개, 챕터 2~4는 각 5개 맵으로 구성한다.
- 같은 성취기준을 여러 맵에서 다룰 때는 수치만 바꾸지 않고 직무 상황, 정보 표현, 조작과 사고 단계를 달리한다.
- 챕터 1은 `도입 → 연습 → 종합`, 챕터 2~4는 `도입 → 연습 → 적용 → 심화 → 종합`의 흐름으로 난이도를 높인다.
- 각 챕터의 마지막 맵은 해당 단원의 여러 학습 경험을 연결하는 종합 맵으로 설계하는 것을 우선 검토한다.
- 뒤의 챕터에서는 이전 챕터의 개념을 보조적으로 다시 사용할 수 있다. 단, 현재 챕터의 주 단원과 학습 목표가 흐려지지 않아야 한다.
- 챕터 전환은 단순한 화면 구분이 아니라 등굣길의 장소, 사건과 WH의 경험이 달라지는 이야기 전환점이 되어야 한다.
- 챕터 4의 마지막 맵은 학교 도착과 수학의 필요성 및 유용성에 대한 WH의 깨달음으로 이어져야 한다.

## 4. 전체 제작 흐름

### 단계 0. 프로젝트 기준 확정

**목표**  
프로젝트의 방향과 변경하면 안 되는 핵심 조건을 분명히 한다.

**주요 작업**

- 대상 학생과 교육 목표 확인
- 4개 단원과 18개 성취기준 확인
- 집에서 학교까지 이어지는 이야기 구조 확인
- WH의 캐릭터 방향 확인
- 2D 싱글 플레이 웹 게임이라는 범위 확인
- GitHub 관리 및 Vercel 배포 방식 확인

**산출물**

- `agent.md`
- `workflow.md`
- 미정 사항 목록

**통과 조건**

- 새 기획이 기존 확정 사항과 충돌하는지 판단할 기준이 있다.

### 단계 1. 18개 맵 교육·서사 설계

**목표**  
18개 맵의 역할과 난이도 흐름을 구현 전에 한눈에 확인한다.

**주요 작업**

1. 집에서 학교까지의 이동 경로를 18개 장소로 나눈다.
2. 이동 경로를 4개 챕터로 나누고 챕터 1에 3개 장소, 챕터 2~4에 각 5개 장소를 배치한다.
3. 각 챕터에 해당 단원과 어울리는 중심 사건 및 이야기 전환점을 정한다.
4. 각 장소에 자연스럽게 연결되는 직업 현장 또는 직무 상황을 정한다.
5. 각 맵에 주 성취기준 1개와 필요한 경우 보조 성취기준을 배정한다.
6. 각 단원의 성취기준이 해당 챕터 안에서 충분히 다뤄지는지 확인한다.
7. 18개 성취기준의 등장 횟수와 순서를 확인한다.
8. 각 맵의 앵커 사건, 핵심 수학 행동, 게임 조작, 성공 결과를 정한다.
9. 챕터 내부와 전체 18개 맵의 난이도 곡선을 함께 검토한다.
10. 앞선 맵이나 챕터에서 배운 내용이 뒤에서 다시 쓰이는 지점을 표시한다.

**산출물**

- 18개 맵 마스터 표
- 4개 챕터 구성표와 챕터별 이야기 흐름
- 전체 등굣길 지도 초안
- 성취기준 배치표
- 난이도 곡선

**통과 조건**

- 모든 맵의 난관 유형이 서로 구별된다.
- 모든 성취기준이 최소 한 번 이상 의미 있게 다뤄진다.
- 각 챕터의 맵 수가 해당 단원의 성취기준 수와 일치하며, 각 맵이 주 성취기준 하나를 중심으로 구성된다.
- 챕터 안과 챕터 사이의 난이도 상승이 모두 자연스럽다.
- 맵의 순서가 수학적 난이도와 이야기 이동 양쪽에서 자연스럽다.
- 마지막 맵과 결말이 WH의 깨달음으로 연결된다.

### 단계 2. 대표 맵 페이퍼 프로토타입

**목표**  
코드를 작성하기 전에 한 개 맵의 학습과 플레이 흐름을 검증한다.

**대표 맵 선택 기준**

- 정보 탐색, 수학적 판단, 조작, 피드백이 모두 포함된다.
- 직무수학 게임의 특징을 잘 보여 준다.
- 지나치게 단순하거나 후반 통합 문제처럼 복잡하지 않다.
- 짧은 시간 안에 처음부터 끝까지 시험할 수 있다.

**주요 작업**

- 맵 상황과 플레이 목표를 한 문장으로 작성
- 화면 흐름과 이동 경로 스케치
- 플레이어가 발견할 정보 목록 작성
- 핵심 문제와 대안 풀이 설계
- 정답, 대표 오답, 허용 오차 설계
- 4단계 힌트 작성
- 성공 시 세계 변화와 다음 맵 연결 설계
- 종이나 간단한 화면 모형으로 모의 플레이

**산출물**

- 대표 맵 기획서
- 화면 흐름도
- 문제 및 피드백 명세
- 모의 플레이 기록

**통과 조건**

- 플레이어가 긴 설명 없이 목표를 파악할 수 있다.
- 수학이 이야기 진행에 반드시 필요하다.
- 단순 정답 입력 외에 관찰, 선택 또는 조작이 포함된다.
- 오답 원인에 따라 다른 피드백을 줄 수 있다.

### 단계 3. 기술 프로토타입

**목표**  
대표 맵을 실제 브라우저에서 플레이 가능한 형태로 만든다.

**먼저 결정할 기술 항목**

- 웹 프레임워크
- 2D 렌더링 방식 또는 게임 라이브러리
- 키보드, 마우스와 터치 입력 범위
- 맵과 문제 데이터를 저장할 형식
- 저장 방식과 이어하기 범위

**최소 구현 범위**

- 시작 화면
- WH의 이동과 기본 애니메이션
- WH를 따라가며 관심 대상을 함께 보여 주는 카메라
- 환경 및 NPC 상호작용
- 정보 수집과 다시 보기
- 한 개의 완전한 수학 난관
- 정답, 대표 오답, 단계별 힌트와 재시도
- 성공에 따른 환경 변화
- 맵 완료 화면
- 기본 음량 및 글자 관련 설정

**이번 단계에서 제외할 수 있는 것**

- 18개 맵 전체
- 최종 그래픽과 음원
- 복잡한 계정 기능
- 온라인 멀티플레이
- 교육 목표와 무관한 수집 및 꾸미기 기능

**통과 조건**

- 별도의 개발 도구 없이 브라우저에서 실행된다.
- 처음부터 맵 완료까지 진행을 막는 오류가 없다.
- 새로고침이나 재시도 후에도 진행 상태가 비정상적으로 꼬이지 않는다.
- 화면 크기가 달라져도 필수 정보와 조작 요소를 사용할 수 있다.
- 대표 정답, 오답, 경계값 판정이 모두 예상대로 작동한다.

### 단계 4. 프로토타입 검토와 방향 확정

**목표**  
18개 맵 제작 전에 핵심 재미와 학습 효과를 평가한다.

**검토 질문**

- 플레이어는 무엇을 해야 하는지 스스로 알 수 있는가?
- 필요한 정보를 환경에서 찾고 연결하는가?
- 수학이 문을 여는 비밀번호처럼 겉돌지 않는가?
- 조작 난도가 수학적 사고를 방해하지 않는가?
- 오답 피드백을 보고 스스로 수정할 수 있는가?
- 성공했을 때 맵과 이야기가 실제로 변하는가?
- WH의 성격과 귀여움이 행동과 반응에서 드러나는가?
- 다음 맵을 해 보고 싶은 이유가 생기는가?

**결정**

- `통과`: 현재 구조를 기준으로 제작을 확대한다.
- `수정 후 통과`: 문제 흐름, 조작 또는 피드백을 수정하고 다시 검토한다.
- `재설계`: 핵심 게임 루프가 작동하지 않으면 구현 확대를 멈추고 대표 맵을 다시 설계한다.

### 단계 5. 4개 챕터와 18개 맵 제작

맵은 한 번에 모두 만들지 않고 챕터 단위로 제작한다.

- **챕터 1 — 1map1~1map3**: 수와 연산
- **챕터 2 — 2map1~2map5**: 변화와 관계
- **챕터 3 — 3map1~3map5**: 도형과 측정
- **챕터 4 — 4map1~4map5**: 자료와 가능성

각 챕터는 `기획 → 구현 → 교육 검토 → 플레이 검토 → 수정` 순서로 완성한다. 이전 챕터의 문제를 해결하기 전에는 다음 챕터 제작을 확대하지 않는다.

**챕터별 통과 조건**

- 해당 챕터의 모든 맵을 순서대로 완료할 수 있다.
- 각 맵이 해당 챕터의 단원 및 주 성취기준과 직접 연결된다.
- 난관의 조작과 수학적 사고가 반복되지 않는다.
- 난이도가 갑자기 오르거나 내려가지 않는다.
- 새 개념에는 충분한 안내가 있고 익힌 개념에는 과도한 설명이 없다.
- 이전 맵의 정보나 전략이 적절히 다시 사용된다.
- 각 맵 완료가 다음 장소로 이동하는 이야기와 연결된다.

### 단계 6. 전체 통합과 난이도 조정

**목표**  
18개 맵을 하나의 완전한 등굣길 경험으로 연결한다.

**주요 작업**

- 전체 게임을 처음부터 끝까지 연속 플레이
- 성취기준별 문제 수와 깊이 확인
- 맵별 예상 시간과 전체 플레이 시간 확인
- 난이도 급상승 구간 확인
- 반복되는 대사, 조작, 문제 유형 제거
- WH의 성장과 결말을 연결하는 장면 보강
- 저장, 이어하기, 재시작 흐름 확인

**통과 조건**

- 18개 맵의 진행이 끊기지 않는다.
- 후반부가 단순히 수치만 커지는 방식으로 어려워지지 않는다.
- 결말에서 앞선 경험과 수학의 유용성이 자연스럽게 회수된다.

### 단계 7. 품질 검증

검증은 다음 네 관점에서 진행한다.

#### 교육 검증

- 성취기준과 과제가 직접 연결되는가?
- 수치, 단위, 공식과 풀이가 정확한가?
- 여러 풀이가 가능할 때 정당한 방법을 불필요하게 오답 처리하지 않는가?
- 대표 오개념에 맞는 피드백을 제공하는가?

#### 게임 검증

- 목표, 상호작용 대상과 성공 결과가 명확한가?
- 우연한 클릭이나 반복 시도로만 통과할 수 없는가?
- 이동, 카메라와 조작이 안정적인가?
- 진행 불가 상태가 생기지 않는가?

#### 사용자 경험 및 접근성 검증

- 색상 이외의 상태 표시가 있는가?
- 글자와 핵심 수치를 읽기 쉬운가?
- 키보드만으로 필수 진행이 가능한가?
- 카메라 움직임과 시간 제한이 학습을 방해하지 않는가?

#### 기술 검증

- 주요 데스크톱 및 모바일 브라우저에서 실행되는가?
- 화면 크기별로 필수 UI가 잘리지 않는가?
- 오류 메시지와 복구 방법이 있는가?
- 제작용 정보나 비밀값이 배포 파일에 포함되지 않는가?

### 단계 8. 배포와 개선

**배포 순서**

1. 변경 내용을 로컬에서 확인한다.
2. GitHub 저장소에 변경 이력을 남긴다.
3. Vercel 미리보기 배포에서 전체 흐름을 확인한다.
4. 문제가 없을 때 운영 버전으로 배포한다.
5. 배포 주소와 버전별 변경 내용을 기록한다.

**배포 후 확인**

- 첫 화면이 정상적으로 열린다.
- 맵 시작과 완료가 가능하다.
- 정답 판정, 힌트와 저장이 작동한다.
- 브라우저 개발 환경에서 치명적 오류가 발생하지 않는다.
- 새 배포로 기존 진행 데이터가 손상되지 않는다.

## 5. 개별 맵 제작 워크플로

각 맵은 아래 순서로 제작한다.

### 1단계: 맵 정의

- 이 맵에서 WH가 겪는 사건은 무엇인가?
- 누구의 어떤 직무와 연결되는가?
- 플레이어가 마지막에 할 수 있게 되어야 하는 것은 무엇인가?
- 주 성취기준과 보조 성취기준은 무엇인가?

### 2단계: 플레이 흐름 설계

`도착 → 사건 발견 → 정보 탐색 → 수학적 판단 → 실행 → 결과 확인 → 다음 장소 이동`

각 단계에서 플레이어가 보는 것, 하는 것, 생각하는 것을 구분해 기록한다.

### 3단계: 수학 과제 검증

- 주어진 정보만으로 해결 가능한가?
- 불필요한 정보와 필요한 정보가 의도대로 구분되는가?
- 단위와 반올림 기준이 명확한가?
- 정답과 허용 가능한 다른 풀이를 정의했는가?
- 대표 오답이 왜 나오는지 설명할 수 있는가?

### 4단계: 구현

- 맵 환경과 이동 경로
- 상호작용 대상
- 수학 도구와 문제 UI
- 판정 로직
- 힌트와 피드백
- 성공 및 실패 반응
- 다음 맵 연결

### 5단계: 검증

- 정상 풀이
- 허용 가능한 대안 풀이
- 대표 오답
- 최솟값, 최댓값과 경계값
- 반복 상호작용
- 중간 이탈 후 재시작
- 키보드 및 화면 크기별 진행

### 6단계: 완료 기록

- 구현한 성취기준
- 변경된 파일과 기능
- 확인한 테스트
- 알려진 문제
- 다음 맵에서 이어질 내용

## 6. 기능 단위 작업 순서

맵이 아닌 공통 기능도 같은 흐름으로 작업한다.

1. **이해**: 사용자 요청, 관련 문서와 기존 구현을 확인한다.
2. **정의**: 기능의 목적과 완료 조건을 한두 문장으로 적는다.
3. **설계**: 가장 작은 구현 범위와 영향받는 부분을 정한다.
4. **구현**: 기존 구조를 보존하며 필요한 부분만 변경한다.
5. **검증**: 정상 사용, 예외, 접근성과 기존 기능 회귀를 확인한다.
6. **기록**: 결과, 가정, 남은 문제를 문서화한다.

## 7. 사용자와 Codex의 협업 방식

### 사용자가 주로 결정하는 사항

- 교육적 의도와 강조할 성취기준
- 세계관, 캐릭터와 이야기의 최종 방향
- 학생에게 적절한 난이도
- 중요한 시각적 취향
- 프로토타입 및 배포 버전 승인

### Codex가 주로 수행하는 사항

- 결정 내용을 기획 및 기술 문서로 구조화
- 맵과 기능의 프로토타입 구현
- 반복 작업과 콘텐츠 데이터 구조화
- 오류, 계산 판정과 접근성 점검
- GitHub 및 Vercel 배포를 위한 프로젝트 정리
- 선택지가 있을 때 장단점과 추천안 제시

### 확인이 필요한 경우

다음처럼 결과를 크게 바꾸는 결정은 구현을 확대하기 전에 사용자와 확인한다.

- 핵심 게임 조작 방식 변경
- 성취기준의 해석이나 범위 변경
- 전체 맵 순서와 결말 변경
- 새로운 외부 서비스나 개인정보 수집 도입
- 주요 기술 구조 변경

## 8. 문서와 콘텐츠 관리 제안

프로젝트가 커지면 다음 구조로 문서를 나누어 관리한다. 실제 폴더는 기술 프로토타입을 시작할 때 확정한다.

```text
/
├─ agent.md                  프로젝트 원칙과 확정 사항
├─ workflow.md               제작 및 검증 절차
├─ docs/
│  ├─ game-design.md         전체 게임 및 이야기 설계
│  ├─ curriculum-map.md      성취기준과 18개 맵 연결표
│  ├─ character-wh.md        WH 캐릭터 명세
│  ├─ decisions.md           주요 결정과 변경 이유
│  └─ maps/
│     ├─ map-01.md
│     └─ ...
├─ public/                   이미지, 음원과 정적 파일
├─ src/                      게임 코드
└─ tests/                    계산 판정 및 주요 흐름 검증
```

## 9. 맵 상태 표시

각 맵은 다음 상태 중 하나로 관리한다.

- `아이디어`: 핵심 상황만 존재한다.
- `기획 중`: 성취기준과 플레이 흐름을 설계하고 있다.
- `구현 준비`: 기획 검토를 통과해 구현할 수 있다.
- `구현 중`: 실제 플레이 기능을 만들고 있다.
- `검증 중`: 교육, 게임과 기술 검증을 진행하고 있다.
- `완료`: 모든 완료 조건을 충족했다.
- `수정 필요`: 검증에서 해결해야 할 문제가 발견되었다.

## 10. 현재 프로젝트의 다음 작업

1. 4개 챕터의 장소 범위와 중심 사건을 정한다.
2. 각 챕터의 이야기형 이름을 정한다.
3. 챕터 1의 3개 맵과 챕터 2~4의 각 5개 맵을 포함한 18개 맵 마스터 표를 만든다.
4. 각 맵의 장소, 직업 상황, 성취기준과 난관 유형을 배치한다.
5. 챕터 내부와 전체 게임의 난이도 곡선 및 이야기 흐름을 검토한다.
6. 대표 프로토타입 맵 하나를 선택한다.
7. 대표 맵의 페이퍼 프로토타입을 작성한다.
8. 프로토타입에 필요한 웹 기술을 선택한다.
9. 브라우저에서 플레이 가능한 첫 맵을 구현한다.

이 순서는 사용자와의 합의 또는 프로토타입 결과에 따라 수정할 수 있다.
