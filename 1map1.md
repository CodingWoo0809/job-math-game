# 1map1 — 막힌 골목의 계산서

> 제목은 임시 이름이다.  
> 챕터: 1 — 수와 연산  
> 주 성취기준: [12직수01-01]  
> 난이도: 1/5, 안내형  
> 예상 플레이 시간: 6~10분  
> 현재 상태: 플레이 가능 웹 프로토타입 구현·검증 중

## 1. 한 문장 정의

플레이어는 WH와 함께 빵집 식자재의 실제 납품 비용을 계산하고 잘못된 계산서를 바로잡아, 배송차가 막고 있던 등굣길을 연다.

## 2. 학습 목표와 플레이 경험

### 학습 목표

- 직무 상황에서 수량과 단가를 곱해 항목별 비용을 구할 수 있다.
- 여러 항목의 비용과 배송비를 더해 전체 업무 비용을 구할 수 있다.
- 전체 비용을 예산과 비교해 남은 금액을 구할 수 있다.
- 계산 결과를 다른 자료와 비교해 오류가 있는 항목을 찾을 수 있다.
- 같은 결과를 얻는 서로 다른 계산 순서를 비교할 수 있다.

### 플레이 경험

플레이어는 골목을 돌아다니며 주문서, 식자재 상자와 배송 계약서에서 정보를 찾는다. 찾은 정보를 계산 보드에 배치해 실제 비용을 만들고, 계산서의 오류를 수정하면 배송차가 움직이며 다음 길이 열린다.

## 3. 앵커 사건

WH가 집을 나서자 좁은 골목이 빵집 식자재 배송차와 안전 가림막으로 막혀 있다. 차량의 화물 리프트가 펼쳐져 있어 보행자가 지나가면 위험하다.

빵집 담당자와 배송 기사는 납품 계산서의 금액이 주문 예산보다 커서 확인을 멈춘 상태다. 납품 확인이 끝나야 리프트를 접고 차량을 이동할 수 있다. WH는 학교에 가기 위해, 두 사람이 계산서 오류를 찾도록 돕는다.

## 4. 직무 맥락

- **직업 현장**: 동네 빵집의 아침 식자재 입고
- **관련 직무**: 식자재 검수, 구매 비용 확인, 예산 관리, 납품 확인
- **직무 목적**: 주문한 수량과 계약 비용을 확인하고 예산 안에서 정확하게 결제한다.
- **수학이 필요한 이유**: 비용이 틀린 상태에서 납품을 승인하면 빵집의 예산과 재고 기록에 오류가 생긴다.

## 5. 맵 공간 구성

```text
[WH의 집]
    │ 이동 튜토리얼
    ▼
[골목 입구] ── [빵집 담당자 + 주문서]
    │
[식자재 상자 구역] ── [배송차 운전석 + 배송 계약표]
    │
[화물 리프트 + 계산 단말기]
    │ 성공 시 차량 이동
    ▼
[열린 골목 / 다음 맵 방향]
```

### 정보 배치

- **빵집 담당자의 클립보드**: 필요한 품목, 수량과 최대 예산
- **각 식자재 상자의 라벨**: 품목명과 단가
- **배송차 운전석의 계약 스티커**: 계약 배송비
- **계산 단말기**: 현재 청구된 계산서와 항목별 금액

정보를 한 화면에 모두 보여 주지 않는다. 플레이어가 직접 이동하고 상호작용해 필요한 정보를 모으게 한다.

## 6. 문제 데이터

모든 가격은 세금이 포함된 금액으로 설정한다.

| 품목 | 수량 | 단가 | 올바른 항목 비용 |
|---|---:|---:|---:|
| 밀가루 | 3포대 | 18,500원 | 55,500원 |
| 우유 | 4상자 | 12,800원 | 51,200원 |
| 버터 | 2상자 | 27,400원 | 54,800원 |
| 계약 배송비 | 1회 | 6,000원 | 6,000원 |

- **빵집의 최대 예산**: 170,000원
- **식자재 비용 소계**: 161,500원
- **올바른 전체 비용**: 167,500원
- **예산에서 남는 금액**: 2,500원
- **단말기에 표시된 잘못된 청구액**: 172,500원
- **오류 내용**: 배송비가 6,000원이 아닌 11,000원으로 입력됨
- **청구액과 실제 비용의 차이**: 5,000원

## 7. 플레이 흐름

### 장면 1. 집에서 나오기

1. 짧은 시작 장면 후 플레이어가 WH를 움직인다.
2. 화면은 WH를 따라가다가 골목을 막은 배송차와 가림막 쪽으로 부드럽게 이동한다.
3. WH가 가림막에 접근하면 안전상의 이유로 지나갈 수 없다는 표시가 나타난다.

**배우는 조작**: 이동, 관심 대상 확인

### 장면 2. 사건 발견

1. 빵집 담당자와 대화한다.
2. 최대 예산이 170,000원인데 계산서에는 172,500원이 표시되어 있음을 알게 된다.
3. WH의 목표가 `실제 납품 비용을 확인해 계산서 오류 찾기`로 갱신된다.

**배우는 조작**: 대화, 목표 확인

### 장면 3. 정보 탐색

1. 클립보드에서 주문 수량을 확인한다.
2. 실제 상자를 살펴 수량과 단가를 확인한다.
3. 배송차 운전석의 계약 스티커에서 배송비가 6,000원임을 확인한다.
4. 발견한 정보가 WH의 계산 수첩에 기록된다.

**핵심 판단**: 전체 비용을 계산하려면 수량, 단가와 배송비가 모두 필요하다.

### 장면 4. 비용 계산

1. 계산 단말기를 열면 품목 카드, 수량 카드와 연산 기호가 나타난다.
2. 플레이어는 각 품목에 `수량 × 단가` 관계를 만든다.
3. 항목별 비용을 계산해 입력하거나 게임 내 계산기로 구한다.
4. 식자재 비용과 배송비를 더해 전체 비용 167,500원을 구한다.
5. 예산 170,000원에서 전체 비용을 빼 남은 금액 2,500원을 구한다.

### 장면 5. 오류 수정

1. 단말기의 계산서 항목 중 잘못된 줄을 선택한다.
2. 배송비 11,000원을 계약 금액인 6,000원으로 수정한다.
3. 수정 후 전체 비용이 167,500원이고 예산 안이라는 사실을 확인한다.
4. 플레이어가 `납품 확인` 버튼을 누른다.

### 장면 6. 길 열기

1. 빵집 담당자가 수정된 계산서를 승인한다.
2. 배송 기사가 화물 리프트를 접고 차량을 이동한다.
3. 가려졌던 골목과 다음 목적지가 보인다.
4. WH가 작은 성취 동작을 한 뒤 열린 길로 이동한다.
5. 짧은 성찰 문장이 나타난다.

**성찰 문장 초안**: “필요한 수량과 비용을 정확히 계산하니 예산도 지키고 길도 열렸어!”

## 8. 허용할 문제해결 방법

### 전략 A. 항목별 계산

각 품목의 `수량 × 단가`를 구한 뒤 모든 항목과 배송비를 더한다.

`55,500 + 51,200 + 54,800 + 6,000 = 167,500`

### 전략 B. 소계 후 배송비 계산

식자재 세 항목의 소계를 먼저 구한 뒤 배송비를 더한다.

`161,500 + 6,000 = 167,500`

### 전략 C. 잘못된 계산서 직접 비교

항목별 비용이 모두 맞는지 확인하고, 계약 배송비와 청구 배송비의 차이 5,000원을 찾아 청구액에서 뺀다.

`172,500 - 5,000 = 167,500`

세 전략 모두 올바른 근거와 계산을 사용했다면 정당한 풀이로 인정한다.

## 9. 대표 오답과 피드백

| 입력 또는 행동 | 예상 원인 | 피드백 초안 |
|---|---|---|
| 64,700원 | 수량을 곱하지 않고 단가만 더함 | “상자마다 같은 가격이 적용돼. 각 품목의 수량을 단가와 함께 확인해 보자.” |
| 161,500원 | 배송비 누락 | “식자재 비용은 맞아. 납품을 위해 추가되는 계약 비용이 있는지 찾아보자.” |
| 172,500원 | 계산서 값을 검증 없이 사용 | “계산서의 값이 항상 맞는 것은 아니야. 주문서와 계약표의 정보로 직접 확인해 보자.” |
| 2,500원을 전체 비용으로 입력 | 남은 예산과 전체 비용을 혼동 | “이 값은 계산 뒤 남는 돈일까, 실제로 지불할 전체 금액일까?” |
| 식자재 항목을 오류로 선택 | 계약 정보 비교 미완료 | “식자재의 수량과 단가는 주문 내용과 맞아. 별도로 청구된 항목을 살펴보자.” |

오답은 점수나 생명을 잃게 하지 않는다. 해당 정보와 계산 단계를 다시 확인할 수 있게 하고 즉시 재시도하게 한다.

## 10. 단계별 힌트

1. **목표 힌트**: “계산서가 예산을 넘은 이유를 찾으려면 실제로 내야 할 전체 비용을 먼저 구해야 해.”
2. **정보 힌트**: “주문 수량은 클립보드에, 단가는 상자 라벨에 있어. 배송비는 차량 앞쪽에서 확인할 수 있어.”
3. **관계 힌트**: “각 품목은 `수량 × 단가`로 계산하고, 마지막에 배송비를 더해 보자.”
4. **풀이 힌트**: “식자재 비용은 161,500원이야. 계약 배송비 6,000원을 더한 뒤 계산서와 비교해 보자.”

힌트는 첫 오답에 자동으로 모두 보여 주지 않는다. 플레이어가 요청하거나 같은 유형의 오류가 반복될 때 한 단계씩 제공한다.

## 11. 조작과 화면 구성

### 탐색 화면

- 이동: 방향키 또는 WASD를 우선 검토
- 상호작용: 가까운 대상에서 키 입력 또는 화면 버튼
- 발견한 정보: 계산 수첩에서 언제든 다시 확인
- 현재 목표: 짧은 한 문장으로 화면 가장자리에 표시

### 계산 화면

- 품목, 수량, 단가와 연산 기호를 카드 형태로 제공
- 키보드 입력과 화면 숫자 패드를 모두 지원
- 천 단위 쉼표 유무와 관계없이 같은 수로 판정
- 계산 중 단위 `원`을 계속 표시
- 게임 내 계산기 사용을 허용하되, 어떤 수를 왜 계산하는지는 플레이어가 구성하게 함

구체적인 입력 체계는 기술 프로토타입에서 확정한다.

## 12. 카메라와 연출

- 평상시 카메라는 WH의 이동을 부드럽게 따라간다.
- 골목에 처음 도착하면 WH를 화면 한쪽에 두고 배송차, 가림막과 빵집 담당자를 함께 보여 준다.
- 새 정보가 발견되면 해당 대상과 WH의 반응을 동시에 볼 수 있는 구도를 우선한다.
- 계산 중에는 세계를 완전히 가리지 않고 배송 현장이 배경에 남게 한다.
- 성공 시 카메라는 차량 이동과 열린 길을 먼저 보여 준 뒤 WH에게 돌아온다.

## 13. 대사 초안

**WH**: “학교 가는 길이 차로 꽉 막혔네.”

**배송 기사**: “리프트가 펼쳐져 있어서 지금은 지나가면 위험해. 그런데 계산서 확인이 끝나질 않네.”

**빵집 담당자**: “예산은 17만 원인데 청구액은 17만 2천5백 원이야. 주문한 수량과 계약 배송비를 같이 확인해 줄래?”

**WH**: “주문서와 실제 상자를 비교한 다음, 전체 비용을 계산해 볼게요.”

**빵집 담당자, 성공 후**: “배송비 입력이 잘못됐구나. 덕분에 예산 안에서 정확히 처리했어!”

**배송 기사**: “확인 완료! 바로 길을 비켜 줄게.”

대사는 기능 설명을 장황하게 반복하지 않으며 최종 문구는 전체 캐릭터 말투가 정해진 뒤 다듬는다.

## 14. 접근성

- 핵심 정보는 색상뿐 아니라 글자, 아이콘과 위치로 구분한다.
- 가격과 수량을 음영이 복잡한 배경 위에 직접 표시하지 않는다.
- 대사는 자막으로 제공하고 플레이어가 넘기기 전까지 유지한다.
- 계산 화면은 키보드만으로도 조작할 수 있게 한다.
- 시간 제한을 두지 않는다.
- 잘못 입력해도 언제든 수정하고 다시 시도할 수 있게 한다.

## 16. 성공·실패 상태

### 성공

- 계산서가 167,500원으로 수정된다.
- 빵집 예산이 2,500원 남는다.
- 배송차와 가림막이 이동한다.
- 다음 맵으로 가는 길이 열린다.
- WH의 수첩에 `[비용은 수량과 단가를 확인한 뒤 모두 더하고, 예산과 비교한다]`가 기록된다.

### 진행 중 오류

이 맵에는 영구 실패가 없다. 잘못 계산하거나 항목을 선택하면 원인에 맞는 피드백을 받고 정보 탐색 또는 계산 단계로 돌아간다.

## 17. 완료 조건

- 집 앞에서 열린 골목까지 전체 흐름을 플레이할 수 있다.
- 주문서, 상자 라벨과 계약표를 직접 탐색해야 한다.
- 항목별 비용, 전체 비용과 남은 예산을 올바르게 판정한다.
- 세 가지 문제해결 전략을 정당한 풀이로 수용한다.
- 대표 오답별 피드백과 4단계 힌트가 작동한다.
- 성공하면 배송차가 움직이고 다음 길이 열린다.
- 이동, 대화와 계산을 키보드만으로도 완료할 수 있다.

## 18. 추후 결정할 사항

- 최종 맵 제목
- WH와 NPC의 확정 말투
- 정확한 이동 및 상호작용 키
- 계산 수첩과 단말기의 시각 디자인
- 게임 내 계산기 제공 방식
- 빵집과 배송차의 최종 그래픽
- 1map2로 이어지는 구체적인 장소와 사건

## 19. 구현 현황

- 2D 횡스크롤 이동과 WH 추적 카메라
- 사건 발견, NPC 대화와 목표 갱신
- 주문서, 품목 라벨 및 계약 배송비 탐색
- 발견 정보를 다시 확인하는 계산 수첩
- 전체 비용과 예산 잔액 판정
- 계산서 오류 항목 선택 및 배송비 수정
- 대표 오답별 피드백과 단계별 힌트
- 배송차 이동, 길 개방과 맵 완료 연출
- 키보드 및 터치 조작, 큰 글자와 움직임 줄이기 설정
- 브라우저 로컬 저장을 이용한 이어하기
- 계산 결과와 화면 요소 연결에 대한 자동 검증
- 두꺼운 외곽선, 밝은 색감, 통통한 2D 횡스크롤풍 그래픽 시범 적용
